## 🔴 A. Discounted product is charged at REGULAR price

### Root cause — confirmed
`laravel-backend/app/Actions/Orders/PlaceOrder.php:73`

```php
$priceMinor = $product->price->toMinor();   // ← discount_price is never read
$lineMinor  = $priceMinor * $qty;
$subtotalMinor += $lineMinor;
```

`PlaceOrder` is the **only** place a unit price is resolved. `discount_price` appears **zero times** anywhere under `app/Actions/`, `app/Services/`, `app/Support/Payments/`. It exists only on the model, the admin form, and `ProductResource`.

The gateway path is actually watertight — the client never sends money (`StoreOrderRequest.php:13-15`), and SSLCommerz recomputes from the DB (`PaymentAmount.php:28-37` → `orders.total`). That's exactly why the bug is total: the wrong number is baked into `orders.total` at placement, so the invoice, the SMS, the pay link and the gateway **all** charge the regular price.

Storefront shows the discount correctly (`CheckoutForm.tsx:25` — `discount_price ?? price`), so the customer sees Tk 8,000 and is billed Tk 10,000.

Knock-on: the advance/partial payment (`PlaceOrder.php:79-85`) is a % of the inflated line total, so partial payments are overcharged too.

### Fix

1. **Single source of truth on the model** — add to `app/Models/Product.php`:
   ```php
   public function effectivePrice(): Money
   {
       $d = $this->discount_price;
       if ($d !== null && $d->toMinor() > 0 && $d->toMinor() < $this->price->toMinor()) {
           return $d;
       }
       return $this->price;
   }
   ```
   (Guard matters: `discount_price` is nullable and un-validated against `price` today — a bad admin entry could otherwise *raise* the price.)
2. `PlaceOrder.php:73` → `$priceMinor = $product->effectivePrice()->toMinor();`
3. `ProductResource.php:38-39` → also expose `effective_price`, and switch the storefront to it (`CheckoutForm.tsx:25`, `ProductActions.tsx:20`, `ProductRow.tsx:79`, `HeaderSearch.tsx:123`, `product/[slug]/page.tsx:41`, `lib/whatsapp.ts:36,56`) so preview and charge can never drift again.
4. **Snapshot the saving.** `order_items` has only `price/qty/line_total` (`create_order_items_table.php:20-22`) — the original price is lost. Migration: `order_items.original_price` (unsignedBigInteger, nullable) + `order_items.discount_amount`. Fill both in `PlaceOrder.php:103-110`.
5. **Order-level discount columns** (needed by item D too): `orders.discount` + `orders.discount_note` (see D). `total = subtotal − discount + shipping` becomes the one formula (`PlaceOrder.php:163`).
6. Add validation on the product form: `discount_price` must be `< price` (`ProductFormRequest`).

### Tests (Pest)
- `PlaceOrderTest`: product with `discount_price` → `order_items.price` == discount, `orders.subtotal`/`total` reflect it.
- discount_price ≥ price → falls back to regular price.
- `discount_price = 0`/null → regular price.
- Partial/advance amount computed on the discounted line total.
- `PaymentAmount::for()` on a discounted order returns the discounted payable.

### Backfill
Existing orders keep their (wrong) totals — **do not** retro-edit paid orders. Only flag them: a one-off report of unpaid/pending orders where `order_items.price != product.effectivePrice()`, so the owner can decide per order (and now, with item D, apply a manual discount).

## 🟢 B. Multi-product / bundle shipping charge (zone-aware discount)

### Current state — better than expected
Zone-aware per-product shipping **already exists**: `product_shipping_charges(product_id, shipping_zone_id, extra_cost)` (`2026_06_27_000001_…:13-23`), admin UI at `catalog/products/form.tsx:629-670`, lookup `Product::extraPerUnitMinorFor()` (`Product.php:87-98`).

Formula today — `PlaceOrder.php:130-148`:
```
shipping = (any chargeable line ? zone.cost : 0)
         + Σ over chargeable lines [ extra_cost(product, zone) × qty ]
```
The zone base is charged **once per order**; the per-product extra is **per unit × qty**. Mirrored client-side in `CheckoutForm.tsx:50-59` (⚠ formula duplicated in PHP + TS — drift risk).

### What's actually missing
The owner wants: *"when a customer buys more than one of a product (or several products), reduce the per-product shipping."* Today qty 3 = 3× the extra charge, which is wrong for furniture (one truck, not three).

### Fix — add a **quantity-tiered / bundle rule** per product-zone

New table `product_shipping_tiers`:
| col | type |
|---|---|
| `product_id` FK | |
| `shipping_zone_id` FK | |
| `min_qty` | unsignedSmallInteger |
| `extra_cost` | unsignedBigInteger (paisa, **per unit at this tier**) |
| unique | `(product_id, shipping_zone_id, min_qty)` |

Semantics: pick the row with the **highest `min_qty` ≤ line qty**; fall back to `product_shipping_charges.extra_cost` (= the `min_qty = 1` tier). Example: chair extra ৳300/unit; tier `min_qty=2 → ৳150`; tier `min_qty=4 → ৳0`. Buy 3 → 3 × ৳150.

Also add a **per-order cap**: `products.shipping_charge_mode` enum (`per_unit` | `per_line_flat`) so a bulky item can charge its extra once per line instead of per unit.

### Critical refactor (do this first)
**Extract `PlaceOrder.php:130-148` into `app/Services/Shipping/ShippingCalculator.php`** with a single public method:
```php
public function for(Collection $lines, ?ShippingZone $zone): ShippingBreakdown
```
Then:
- `PlaceOrder` calls it.
- The new **admin order create** (item G) calls it.
- The new **admin order edit/discount** (item D) recalculates through it.
- A new **`POST /api/v1/shipping/estimate`** (`{items[], shipping_zone_id}`) calls it — and the storefront `CheckoutForm` **stops computing shipping in TS** and just renders what the server returns. That permanently kills the PHP/TS drift.

### Admin UI
`catalog/products/form.tsx` — under the existing per-zone extra grid, an "Advanced: quantity tiers" repeater per zone (`min_qty` + `extra ৳`). Sync in `ProductUiController::syncShippingCharges` (`:368-405`).

### Tests
`ShippingCalculatorTest` — tier boundaries (qty 1/2/3/4), no-tier fallback, `shipping_charge_allowed=false`, free-shipping product mixed with a charged one, zone base charged once, multi-product cart.

   - Reject if `payment_status = paid` (or require an explicit "issue refund instead" path).

Ship v1 with a hard block + a clear message.
   - Audit-log the before/after and the note.
4. **Pay link auto-adjusts for free** — `PaymentAmount::for()` (`PaymentAmount.php:28-37`) reads `orders.total` and `advance_paid` live, and `SslController.php:44` recomputes on every init. Nothing to change. ✅ (Confirm with a Pest test.)
5. **Invoice auto-adjusts** — `GenerateInvoicePdf` renders from the order snapshot; add a "Discount" row to `resources/views/invoices/_document.blade.php` (and the shipping label if the COD amount is printed there).
6. **UI** — `orders/show.tsx` totals card: a "Apply discount" section (amount in ৳ + required note), showing the recomputed Total / Paid / Due, and a "Send updated pay link" button (item E).

## 🟢 E. Pay link visible + copyable in the admin order view

### Current state
`app/Support/Orders/PayLink.php:18-33`:
```php
$base.'/pay/'.$order->order_no.'?t='.hash_hmac('sha256', 'pay:'.$order->order_no, config('app.key'))
```
It is **not stored** on the order and is **only** used inside `MessageTemplate::forOrder()` (`:47`) for the SMS `{pay_url}` placeholder. It is never sent to the admin Inertia payload, so there is nothing for the admin to copy.

### Fix
1. `OrderController::show()` → add `'pay_url' => PayLink::for($order)` to the payload.
2. `orders/show.tsx` → a "Payment link" row: the URL (truncated) + a **Copy** button (`navigator.clipboard.writeText`) + "Open" + "Send SMS again".
3. **Resend SMS**: `POST /admin/orders/{order}/resend-pay-link` → dispatches `SendOrderNotification` for the `Placed` event. ⚠ The idempotency guard (`BaseOrderNotificationChannel::alreadySent():70`, unique on `notification_logs(order_id, event, channel)`) will silently swallow a naive re-dispatch — the endpoint must either delete that log row or use a distinct `pay_link_resent` event. Rate-limit it (e.g. 3/hour/order) so it can't be used to spam an SMS bill.

### 🔒 Security note (must fix while we're here)
The pay token is a **plain HMAC of the order number with no expiry and no revocation**. Anyone who ever sees the link (SMS forward, screenshot, browser history) can reopen it forever. Recommended hardening — do it now, it's cheap:
- Add `orders.pay_token` + `orders.pay_token_expires_at`; generate a random 32-byte token at placement, store the **hash**, expire in e.g. 14 days, and allow the admin to **regenerate** (which invalidates the old link).
- Keep `PayLink::verify()` as a hash comparison against the stored hash (`hash_equals`).
- The pay page already only exposes a summary (`PayPageController.php:31`) — but with an unexpirable token, that summary (name, phone, address, total) is permanently readable by anyone with the URL. That's a real PII leak.

## 🔴 G. Courier API Connect broken on the server

### Ranked root causes (from the code)

**1. The keys were saved in the WRONG PLACE.** (most likely)
There are **two** credential stores:
- `couriers.config` — `encrypted:array` cast (`Courier.php:70-78`), edited at **Shipping → Couriers** (`CourierUiController`, `routes/web.php:193-199`).
- the legacy `steadfast` **settings** group — edited at **Settings → Integrations** (`IntegrationSettingController.php:100-113`, `settings/integrations.tsx:228-287`).

The booking **gate** is `Courier::isConfigured()` (`Courier.php:125-138`), which reads **only** `couriers.config`. It does not know about the legacy fallback that the driver factory actually uses (`RepositoryServiceProvider.php:85-86`). So keys typed into **Settings → Integrations** leave `isConfigured() === false`:
- `ShipmentController.php:43-50` refuses to book: *":name is not configured yet."*
- `OrderController.php:96` sends `configured: false` → `orders/show.tsx:393` disables the Book button.
- `OrderObserver.php:54` silently skips auto-push.

And **`docs/PAYMENT-COURIER-HANDOVER.md:77` literally tells the owner to use the wrong page.** The seed migration (`2026_07_05_000004_seed_default_couriers.php:20`) only copies settings→config if the couriers table was empty *at migrate time*, so keys added on the server later never land in `couriers.config`.

**2. APP_KEY mismatch → unhandled `DecryptException` (a hard 500).**
Nothing catches it: `SettingsService.php:60` (`Crypt::decryptString`) and the `encrypted:array` cast (`Courier.php:76`). If the DB was moved, the key rotated, or `bootstrap/cache/config.php` baked a build-time key different from the runtime env, the Integrations page and the Couriers list both 500 on load.

**3. Real provider rejection, hidden behind a useless message.**
`SteadFastCourier.php:28-41` — `POST https://portal.packzy.com/api/v1/create_order` with headers `Api-Key` / `Secret-Key`. It **never checks `$response->successful()`, never logs the body, has no `->timeout()`**. A 401 (bad key / account not API-enabled / server IP not whitelisted) or a 422 (duplicate `invoice` — `order_no` must be unique per merchant, so a re-book after a partial failure fails) all surface as `RuntimeException('Failed to create SteadFast consignment.')`. `ShipmentController::store` has **no try/catch** → Inertia 500 screen.
RedX gotcha: header is `'Bearer '.$token` (`RedxCourier.php:131`) — if the owner pasted a token already starting with `Bearer `, you get `Bearer Bearer …` → 401.

**4. Container egress.** No `timeout`/`retry` on any courier HTTP call → blocked 443 / DNS / TLS gives a `ConnectionException` → same unhandled 500 (or a 504 from Traefik).

**5. Stale Pathao OAuth token.** `PathaoCourier::accessToken()` returns the cached token *before* validating creds (`:174-177`), TTL ~5 days, key `courier:pathao:token:{id}`. Nothing invalidates it when the courier is updated (`CourierUiController::update:65-77`) → 401 after a key change.

**6. Error masking on the location proxy.** `CourierLocationController.php:80-90` — `catch (Throwable)` with **no `report()`**, returns 200 + *"Could not load courier locations. Check the courier credentials and try again."* So every RedX/Pathao failure (including #2 and #4) is invisible in `error_logs`.

**7. Queue worker down.** Auto-push runs in `PushOrderToCourier` — if `queue:work` isn't running, confirmed orders just never book, with no UI signal.

## 🟠 J. WhatsApp inquiry — photo not showing

### Root cause — it can never work as built
`ecommerce-next-frontend/lib/whatsapp.ts:36-48`:
```ts
const photo = imageUrl(product.main_image);
const lines = [
  `*Inquiry — ${product.title}*`,
  `SKU: ${product.sku}`,
  `Price: ${price.formatted}`,
  photo ? `\n${photo}` : "",     // ← a raw .webp file URL, pasted as text
];
return `https://wa.me/${to}?text=${encodeURIComponent(message)}`;
```
Two hard problems:
1. **WhatsApp never attaches an image from `?text=`.** It renders a link *preview* by crawling **OG tags of a page URL** in the text. A bare image-file URL has no HTML/OG tags → no preview. (And the images are **WebP** — `ImageOptimizer.php:35`, `WebpEncoder(quality: 82)` — which WhatsApp's direct-image preview handles poorly anyway.)
2. **`whatsappInquiry` doesn't include the product page link at all.** (`whatsappOrder` at `:65` does; the inquiry one doesn't.) So there's no page for WhatsApp to preview.

Supporting gaps found:
- **No `metadataBase`** anywhere in the Next app (`app/layout.tsx:20-39`).
- Product OG has **no `og:image:width` / `:height` / `og:image:secure_url` / `og:image:type`** (`app/product/[slug]/page.tsx:20-29`).
- **`NEXT_PUBLIC_SITE_URL` is not set** in `.env.local` / `.env.example` → `config.siteUrl` falls back to `http://localhost:3000` (`lib/config.ts:9`), which also poisons the JSON-LD `offers.url` (`product/[slug]/page.tsx:68`).
- The **category page emits no `openGraph` block at all** (`app/category/[slug]/page.tsx:20-23`).

### Fix
1. **Put the product page URL first in the message** (WhatsApp previews the *first* link):
   ```
   *Inquiry — {title}*
   SKU: {sku}
   Price: {formatted}

   {siteUrl}/product/{slug}
   ```
   Drop the raw image URL from the text entirely.
2. **Set `NEXT_PUBLIC_SITE_URL=https://furnib.com`** in the EasyPanel frontend build args (it's a build-time var → requires a frontend redeploy).
3. **Add `metadataBase: new URL(config.siteUrl)`** to `app/layout.tsx`.
4. **Emit a proper OG image** on `app/product/[slug]/page.tsx`: absolute `url`, `secure_url`, `width: 1200`, `height: 630`, `type: 'image/jpeg'`, `alt`.
5. **Generate a JPEG social thumbnail server-side.** WebP is the failure mode here. `ImageOptimizer` already runs on product upload (`ProductUiController.php:15,31`) — add a second derivative: a **1200×630 JPEG** `social_thumbnail`. (The `social_thumbnail` field already exists on the resource — `ProductResource.php:32,51` — it just isn't populated with a WhatsApp-safe format.) Keep it under ~300 KB.
6. Add `openGraph` to the category page too.
7. **Verify with Playwright + the real crawler**: after deploy, run the URL through Facebook's Sharing Debugger (which is the same scraper WhatsApp uses) and force a re-scrape.

```tsx
<div className="relative">

  {clamped && !expanded && (
    <button className="absolute right-0 bottom-0 flex items-center
                       bg-gradient-to-l from-surface from-40% to-transparent pl-10
                       text-sm font-semibold text-accent">
      … See more
    </button>
  )}
</div>
```
The mask fails because the **card background is `bg-surface/30`** (`ProductRow.tsx:44` — 30% opacity) while the gradient is a fully-opaque `from-surface`, and the `to-transparent` half is see-through by design. So the clamped words bleed through the button. Nothing reserves vertical space — the container is only `min-h-[2.6em]` (exactly 2 lines), with no third row and no gap.

### Fix — exactly what the owner asked for
Stop overlaying. Two clamped lines with a real ellipsis, then the button as a **normal block sibling on its own line, with a gap**:
```tsx
<div className="flex flex-col gap-1.5">
  <p className={expanded ? "whitespace-pre-line …" : "line-clamp-2 …"}>{text}</p>
  {clamped && !expanded && (
    <button type="button" onClick={expand}
            className="self-start text-sm font-semibold text-accent underline-offset-2 hover:underline">
      See more
    </button>
  )}
</div>
```
- `line-clamp-2` already renders the `…` at the end of line 2 — that satisfies *"first line full, second line half then `…`"*.
- `gap-1.5` gives the requested breathing room.
- `self-start` puts it left-aligned on its own line.
- Remove `min-h-[2.6em]` and the gradient entirely; instead reserve card height with a fixed caption block so the grid doesn't jump (`min-h` on the wrapper, not the `<p>`).
- Tap target: give the button `py-1` so it clears the 44px guideline together with its line-height.

**Verify with Playwright** at 360px / 390px / 414px widths on `/category/{slug}`, with short (1-line), exactly-2-line, and long descriptions.

## 🟠 L. Desktop category banner cropped

Reference: https://furnib.com/category/revolving-tulip-chair

### Root causes (three, all real)
`ecommerce-next-frontend/app/category/[slug]/page.tsx:43-65`:
```tsx
<section className="relative mt-3 h-[22vh] w-full overflow-hidden rounded-card border border-border sm:h-[50vh]">
  <picture className="block h-full w-full">
    <source media="(min-width:768px)" srcSet={desktopHeader} />
    <img src={mobileHeader ?? desktopHeader} className="h-full w-full object-cover" />
  </picture>
```
1. **It's a viewport-height box, not an aspect box.** `h-[22vh] sm:h-[50vh]` + `object-cover` means the container's aspect ratio changes with the *window height*. On a 1920×900 screen the box is ≈4.3:1; on a tall monitor ≈2.7:1. The uploaded banner is 1600×600 (2.67:1) → `object-cover` crops top/bottom by an amount that varies per visitor. There is no `aspect-ratio` anywhere.
2. **Breakpoint mismatch.** The `<source>` swaps at `min-width:768px` (`md`) but the height class swaps at `sm:` (640px). **Between 640–767px the portrait mobile image (800×1000) is stretched into a 50vh landscape box** → severe crop.
3. **No server-side processing for category banners.** `CategoryUiController.php:109-118` stores the raw upload — `ImageOptimizer` is wired into `ProductUiController` only. `CategoryFormRequest.php:38-39` validates mime + 20 MB, **no dimension rules**. The "recommended 1600×600" text in `catalog/categories/form.tsx:159-176` is just copy — nothing enforces it.

### Fix
1. **Match the box to the asset**: replace the `vh` heights with an aspect box —
   ```tsx
   <section className="relative mt-3 w-full overflow-hidden rounded-card border border-border
                       aspect-[4/5] md:aspect-[8/3]">
   ```
   (`8/3` = 2.67:1, exactly the 1600×600 recommendation; `4/5` = 800×1000 for the mobile portrait asset.) Then `object-cover` has nothing left to crop.
2. **Align the breakpoints** — move the height/aspect switch to `md:` so it matches the `<source media="(min-width:768px)">`.
3. **Enforce the aspect on upload**: add `ImageOptimizer` to `CategoryUiController` and **resize/pad to exactly 1600×600 (desktop) and 800×1000 (mobile)** — `cover`-fit crop from the centre, or better, add a small crop UI. At minimum add `dimensions:min_width=1600,min_height=600` to `CategoryFormRequest` and show a live aspect-ratio preview in the admin form.
4. Add `og:image` for category pages while touching this file (see item J).

**Verify with Playwright** at 1280×720, 1440×900, 1920×1080 and a tall 1440×2560, plus 700px (the broken 640–767 band), against the real banner.

## 🟢 M. Facebook Commerce Management page

### Current state
A **public, unauthenticated** CSV feed already exists: `routes/web.php:39` → `FeedController@products` → `app/Services/Marketing/ProductFeed.php`. It emits 11 columns:
`id, title, description, availability, condition, price, sale_price, link, image_link, additional_image_link, brand`
Price is already formatted `"1234.00 BDT"`, `id` = SKU (correctly matched to the Pixel `content_ids`), `sale_price` carries `discount_price`. That's a solid base — but it's thin, unfiltered, and unprotected.

### 🔒 Security issues to fix first
- **The feed is public.** Anyone can scrape your full catalogue with prices and stock at `https://admin.furnib.com/feed/products.csv`. Meta supports **HTTP Basic auth on scheduled feed URLs** — put it behind Basic auth + an unguessable path segment, credentials stored encrypted, and rate-limit it.
- The **Meta System User token** must be stored **encrypted in the DB** (`encrypted` cast, like `couriers.config`), never in `NEXT_PUBLIC_*`, never in git. All Graph calls server-side only. Never log the token or a request body containing it.
- ⚠️ **Unrelated but urgent:** `docs/sms-gateway/notes.txt` and `code-examples.txt` are **untracked in git right now** and contain a **real Automas SMS API key**. Do **not** `git add` them. Add them to `.gitignore` and **regenerate that key** — treat it as leaked.

### Plan
**1. Extend `ProductFeed`** to the full Meta spec (verified against [Meta catalog fields](https://developers.facebook.com/docs/commerce-platform/catalog/fields/)):

```
id, title, description, availability, condition, price, sale_price,
sale_price_effective_date, link, image_link, additional_image_link, brand,
item_group_id, google_product_category, fb_product_category, product_type,
quantity_to_sell_on_facebook, status, color, material, shipping, shipping_weight,
mpn, custom_label_0..4
```
Rules that matter:
- `availability`: only **`in stock`** / **`out of stock`** (Meta consolidated the old values).
- `sale_price` must be **strictly less** than `price` — never emit `sale_price == price`.
- `sale_price_effective_date`: ISO-8601 range with **`+06:00`** so Dhaka-time promos start/stop correctly.
- `image_link`: JPEG/PNG, **min 500×500**, ≤8 MB. ⚠️ **Our images are WebP** — reuse the JPEG derivative from item J.
- `product_type` = the category breadcrumb (`Home > Living Room > Sofas`) — this is what powers feed filtering.
- Never `DELETE` a product on stock-out — flip `availability`; deleting destroys retargeting history. Use `status: archived` for discontinued.
- `id` is **immutable forever** — it's the join key for Pixel, CAPI, ads reporting and DPA.

**2. DB**
- `categories.fb_product_category` + `categories.google_product_category` (mapping table driven from Meta's downloadable taxonomy; a dropdown per category in the admin).
- `settings` group `facebook_commerce`: `catalog_id`, `business_id`, `system_user_token` (encrypted), `feed_username`, `feed_password` (encrypted), `graph_version`.
- `meta_catalog_sync_batches`: `handle`, `status`, `item_count`, `errors_json`, `finished_at`.

**3. API sync — `POST /{catalog_id}/items_batch`** (the old `/{catalog_id}/batch` is deprecated):
```json
{ "item_type": "PRODUCT_ITEM", "allow_upsert": true,
  "requests": [ { "method": "UPDATE", "data": { "id": "SKU-1", "title": "…",
    "price": "52500.00 BDT", "sale_price": "44900.00 BDT", "availability": "in stock", … } } ] }
```
- `allow_upsert: true` + always `method: UPDATE` = idempotent upserts.
- **≤5,000 items/request** (Meta recommends ≤3,000), payload ≤28 MB → chunk at 1,000.
- Rate limit ~100 calls/hour/catalog (error `80014`) → read `X-Business-Use-Case-Usage` and back off.
- It's **async**: a 200 does not mean ingestion succeeded. Persist the returned `handle` and poll
  `GET /{catalog_id}/check_batch_request_status?handle=…&load_ids_of_invalid_requests=true`.

**4. Laravel layering** (per CLAUDE.md): `MetaCatalogClient` (HTTP) → `MetaCatalogService` → `ProductToCatalogItemMapper` (spatie/laravel-data DTO) → queued `SyncCatalogItemsJob`. Trigger on product save / price change / stock change (debounced), plus a nightly full reconciliation through the feed URL so drift self-heals.

**5. Admin page** — sidebar **Marketing → Facebook Commerce** (the Marketing group already exists, `app-sidebar.tsx:81-84`):
- **Connection** card: Catalog ID, Business ID, System User token (write-only, masked), "Test connection" (`GET /{catalog_id}?fields=name,product_count`).
- **Feed** card: the Basic-auth-protected feed URL + Copy, credentials, "Regenerate", "Download CSV now", and the exact steps to add it as a scheduled feed in Commerce Manager (min interval = **1 hour**; ≤1M items).
- **Export** card: filter by **category (multi-select) / status / stock / all** → download CSV.
- **Sync** card: "Sync all" / "Sync selected categories" → shows batch handles + live status from `check_batch_request_status`, with per-item errors/warnings.
- **Health check**: sample N products and assert the storefront's emitted Pixel `content_ids` **exactly equal** the feed `id`s — a mismatch here is the #1 reason catalog ads underdeliver.

**6. Owner setup (one-time, in Meta)**
Business Manager → Commerce Manager → **Create catalog** (E-commerce) → note the **Catalog ID** → Meta for Developers → **Create Business App** → Business Settings → **System Users** → add `furnib-catalog-sync` → **Add Assets → Catalogs → Full control** → **Generate token** with **`catalog_management`** + `business_management` → paste into the admin page.

**7. 🇧🇩 BD caveat — scope the page correctly.** Native Meta **checkout is not available in Bangladesh**, so the catalog is for **ads (Advantage+ / DPA), Instagram tagging, and WhatsApp/Messenger product sharing** — the buy action links out to the storefront. So **`link` correctness matters more than `quantity_to_sell_on_facebook`**, and we should **not** build Shops Order Management sync (unusable in BD).
